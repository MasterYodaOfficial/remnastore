# Rollback Checklist

Этот чеклист описывает минимальный rollback ritual для неудачного релиз-кандидата.

## Когда откатываемся

- [ ] Появились `5xx` или auth regressions в критичных user flows
- [ ] Сломались платежи, webhook, account linking или withdrawals
- [ ] Admin не может выполнять обязательные операционные действия
- [ ] Релиз несет security risk или утечку чувствительных данных

Если хотя бы один пункт подтверждается, rollback не откладываем "до утра".

## Перед rollback

- [ ] Зафиксирован commit/tag, на который откатываемся
- [ ] Понято, затрагивались ли миграции БД и обратимы ли они
- [ ] Сохранены логи, traceback и краткое описание симптома
- [ ] Ответственный понимает, нужен rollback только кода, только env или вместе с данными

## Rollback по слоям

### Web/Admin

- [ ] Вернуть предыдущий стабильный frontend build
- [ ] Очистить CDN/cache, если это нужно для статики
- [ ] Проверить browser и Mini App boot после отката

### API/Bot/Workers

- [ ] Вернуть предыдущий стабильный image/commit для `api`, `bot`, `worker`, `notifications-worker`
- [ ] Перезапустить сервисы контролируемо, не смешивая новый и старый runtime
- [ ] Проверить health-check и ключевые endpoints после отката

### Env / Config

- [ ] Вернуть предыдущий набор env-переменных, если регрессия вызвана конфигом
- [ ] Сверить rollback-состояние с [`docs/production-env.md`](./production-env.md)

### Database

- [ ] Не откатывать миграции автоматически, пока не подтверждена их обратимость
- [ ] Если миграция необратима или уже затронула данные, остановиться и переходить к ручному recovery plan
- [ ] При необходимости использовать подтвержденный backup/restore сценарий вместо импровизации

## После rollback

- [ ] Повторить критичный smoke по [`docs/smoke-checklist.md`](./smoke-checklist.md)
- [ ] Проверить, что платежи, webhook и уведомления снова в норме
- [ ] Открыть follow-up задачу с root cause и условиями повторного релиза
- [ ] Новый релиз не делать, пока не зафиксирован remediation plan
