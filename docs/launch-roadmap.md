# Дорожная карта запуска

## Назначение

Документ фиксирует путь от текущего состояния репозитория до первого коммерческого запуска RemnaStore.

Практический прогресс по фазам ведется в [`docs/launch-progress.md`](launch-progress.md).

В рамки этого запуска входят:
- Telegram-бот
- Telegram Mini App
- браузерное приложение
- платежи через YooKassa и Telegram Stars
- кошелек и неизменяемый ledger
- рефералка с настраиваемой ставкой и заявками на вывод
- уведомления в Telegram и внутри приложения
- отдельная легкая админка
- разовая миграция пользователей из старого бота

## Что считаем готовым запуском

Запуск считается готовым, когда выполнены все условия:
- пользователь может войти через Telegram или browser auth и связать обе идентичности в один локальный аккаунт
- пользователь может активировать trial один раз с anti-abuse ограничениями
- пользователь может пополнить баланс через YooKassa и Telegram Stars
- пользователь может купить тариф либо с баланса, либо напрямую при оплате
- backend создает или обновляет пользователя в Remnawave и возвращает рабочий `subscription_url`
- все денежные операции пишутся в неизменяемый ledger
- рефералы считаются корректно, награда начисляется при первой успешной покупке и видна в кабинете
- реферальный заработок можно вывести через заявку с ручной обработкой
- владелец может управлять продуктом через небольшую админку
- стек разворачивается на одном сервере Timeweb через Docker и Nginx

## Текущее состояние репозитория

### Уже есть
- `apps/api` содержит базовую модель аккаунта, auth flow, связку аккаунтов, интеграцию с Remnawave и subscription endpoints
- `apps/bot` уже содержит Telegram-бота и webhook-сервер
- `apps/web` уже содержит пользовательский browser app и Telegram Mini App shell
- связка browser <-> Telegram реализована в новом стеке
- баланс уже хранится в рублях как целое число

### Еще не закончено
- обработка платежных webhook в новом API пока заглушка в `apps/api/app/api/v1/endpoints/webhooks.py`
- нет полноценного ledger-модуля
- нет абстракции платежных провайдеров под несколько шлюзов
- нет моделей и UI для заявок на вывод
- нет отдельного приложения `apps/admin`
- реферальный backend в новом API перенесен не полностью; во frontend еще есть placeholder по оставшейся логике
- в части полей осталось старое именование, например `referral_earnings_cents`, хотя баланс уже хранится в рублях

## Архитектурные решения

### 1. API остается источником истины
- состояние денег, рефералов, уведомлений и подписок живет в проектной БД
- Remnawave остается системой провижининга, а не биллинга
- Supabase остается auth provider, а не источником бизнес-состояния

### 2. Кошелек и неизменяемый ledger вводим сразу
Не продолжаем модель с простым изменяемым балансом без журнала.

Подход:
- `accounts.balance` остается кэшем текущего баланса
- каждое изменение денег создает запись в ledger
- баланс меняется только через доменные сервисы, которые пишут ledger entries

Типы событий ledger для v1:
- зачисление оплаты
- ручное зачисление
- ручное списание
- списание за покупку тарифа
- фиксация выдачи trial
- реферальное начисление
- резервирование суммы на вывод
- возврат резерва при отмене вывода
- финальное списание при успешной выплате
- возврат средств
- корректировка

### 3. Перед интеграцией шлюзов делаем слой платежных провайдеров
Нужен единый внутренний интерфейс:
- `PaymentGateway`
- `YooKassaGateway`
- `TelegramStarsGateway`
- `PaymentService`

Это не даст provider-логике расползтись по API routes и упростит добавление новых шлюзов.

### 4. Админку выносим в отдельное небольшое приложение
Рекомендуемая структура:

```text
apps/
  admin/   отдельный UI для админки
```

Причины:
- отдельная авторизация
- отдельная навигация и сценарии
- меньше риск смешать admin и user flows в одном приложении

### 5. Прямая покупка и покупка с баланса должны идти через один purchase service
Даже если пользователь покупает напрямую, backend должен доводить итог через один и тот же внутренний purchase service и одни правила ledger.

Так pricing, trial rules, referral rules, продление подписки и учет денег останутся согласованными.

## Модели данных, нужные для запуска

### Admins
Минимальная таблица:
- `id`
- `email`
- `password_hash` или внешний auth identifier
- `is_active`
- `created_at`
- `last_login_at`

Если получится, лучше использовать browser auth плюс локальную allowlist-привязку, а не обычный пароль. Но для v1 маленькая локальная admin auth тоже допустима.

### Ledger
Обязательные поля:
- `id`
- `account_id`
- `entry_type`
- `direction`
- `amount_rub`
- `currency`
- `balance_before`
- `balance_after`
- `reference_type`
- `reference_id`
- `comment`
- `created_by_admin_id`
- `created_at`

Правило: строки append-only.

### Payments
Нужны сущности:
- платежный провайдер
- payment intent / invoice
- provider callback
- idempotency key
- snapshot payload провайдера
- история статусов платежа

Правило: каждый callback от провайдера должен быть идемпотентным.

### Referrals
Рекомендуемые сущности:
- атрибуция реферала к аккаунту
- отдельные партнерские ссылки или кампании
- записи реферальных наград, связанные с ledger
- глобальная ставка и per-referrer override

### Withdrawals
Обязательные поля:
- `id`
- `account_id`
- `amount_rub`
- `destination_type` (`card`, `sbp`)
- `destination_value`
- `user_comment`
- `admin_comment`
- `status`
- `created_at`
- `updated_at`
- `processed_by_admin_id`

Правило: выводить можно только реферальные начисления, а не весь баланс кошелька.

### Notifications
Обязательные поля:
- `id`
- `account_id`
- `channel` (`telegram`, `in_app`)
- `type`
- `title`
- `body`
- `is_read`
- `created_at`
- `sent_at`

## Бизнес-правила v1

### Аккаунты и trial
- один trial на локальный аккаунт
- anti-abuse должен смотреть и на linked auth identities, и на Telegram identity
- trial нельзя активировать после наличия платной подписки
- заблокированный пользователь не может запускать trial, создавать платежи и отправлять заявки на вывод

### Тарифы и подписки
- поддерживаются планы `1`, `3`, `6`, `12` месяцев
- длительность trial должна настраиваться из admin или системных настроек
- базовый запуск строится вокруг `subscription_url`
- готовые client links допустимо отложить, если они задерживают запуск
- каждая покупка должна завершаться через один internal subscription activation service

### Платежи
- доступны и direct purchase, и wallet purchase
- успешная оплата либо пополняет кошелек, либо сразу завершает прямую покупку в зависимости от flow type
- дублирующиеся callback не должны создавать двойные зачисления и двойные подписки
- отмененные и истекшие платежи не должны создавать ledger entries, кроме audit trail по статусам
- возвраты должны создавать компенсирующие записи в ledger

### Рефералы
- реферальная награда начисляется только при первой успешной платной покупке
- ставка должна настраиваться глобально
- должен поддерживаться override ставки для конкретного реферера
- self-referral должен быть запрещен
- атрибуция реферала должна происходить только один раз на пользователя
- реферальная награда создается только после успешной финализации платежа

### Выводы
- выводить можно только реферальные начисления
- минимальная сумма вывода должна быть настраиваемой
- статусы: `new`, `in_progress`, `paid`, `rejected`, `cancelled`
- создание заявки должно резервировать доступную сумму
- отклонение или отмена должны возвращать резерв обратно
- успешная выплата должна финализировать списание
- каждое ручное изменение статуса должно сопровождаться комментарием администратора

### Действия администратора
- ручная корректировка баланса только с комментарием
- ручная выдача подписки обязана писать audit record
- блокировка пользователя должна сразу запрещать новые оплаты и новые заявки на вывод

## Фазы реализации

## Фаза 0. Стабилизация текущей базы
Цель: сделать текущий стек надежной основой до внедрения денежных потоков.

Что сделать:
- довести до конца cleanup старых именований вроде `referral_earnings_cents`
- прогнать end-to-end сценарии связки аккаунтов после последних фиксов
- добавить интеграционные тесты на account linking, eligibility trial и subscription sync
- зафиксировать обязательные env-переменные для local и production

Критерий выхода:
- browser и Telegram login/linking работают стабильно
- текущие API и bot поднимаются в Docker без ручных костылей
- базовая подписка через Remnawave работает

## Фаза 1. Биллинг и ledger
Цель: ввести правильную денежную модель до интеграции платежных шлюзов.

Что сделать:
- добавить таблицы ledger и миграцию
- сделать ledger service с атомарными helper-ами обновления баланса
- убрать прямые изменения баланса вне ledger-backed сервисов
- добавить endpoint истории баланса для user app
- добавить админскую корректировку баланса с обязательным комментарием

Критерий выхода:
- любое изменение баланса создает ledger row
- баланс можно восстановить из ledger
- есть идемпотентные тесты на credit и debit операции

## Фаза 2. Платежи: YooKassa и Telegram Stars
Цель: корректно принимать деньги в систему.

Что сделать:
- создать payment abstraction layer
- реализовать YooKassa provider и webhook handling
- реализовать Telegram Stars provider и callback handling
- хранить provider payload и idempotency keys
- поддержать два flow type: `wallet_topup` и `direct_plan_purchase`
- сделать user purchase endpoints и frontend checkout UI

Критерий выхода:
- успешный callback создает один и только один finalized payment record
- top-up меняет баланс через ledger
- direct purchase активирует выбранный тариф без второго параллельного write path

## Фаза 3. Единый purchase flow и Remnawave
Цель: объединить коммерческую логику покупки.

Что сделать:
- сделать один purchase service для trial, direct purchase и wallet purchase
- определить правила продления уже существующей подписки
- писать результат покупки обратно в account state и notifications
- держать Remnawave provisioning за одним integration service

Критерий выхода:
- все активации подписки проходят через один code path
- пользователь всегда получает рабочий `subscription_url`
- ошибка на этапе покупки не оставляет частично записанное billing state

## Фаза 4. Рефералка и выводы
Цель: закончить growth и payout loop.

Что сделать:
- перенести referral logic в отдельные backend services и endpoints
- поддержать referral code attribution и partner overrides
- считать награду только при первой успешной платной покупке
- завести ledger flow для реферальных начислений
- добавить модель, сервис и user UI для заявок на вывод
- добавить admin UI для очереди выводов и обработки статусов

Критерий выхода:
- user app больше не содержит placeholder-состояний по рефералке
- заявки на вывод корректно резервируют и освобождают средства
- администратор обрабатывает выплаты без прямого доступа к БД

- роли в админке на первом запуске могут быть минимальными, если все действия журналируются

## Критерий готовности первого коммерческого запуска
Продукт можно считать готовым, когда новый или мигрированный пользователь может:
- войти из Telegram или браузера
- связать идентичности в один аккаунт
- получить trial при соблюдении правил
- оплатить через YooKassa или Telegram Stars
- купить тариф напрямую или с баланса
- получить активную подписку в Remnawave
- видеть уведомления и реферальное состояние в приложении
- создать заявку на вывод реферальных средств
- перейти в поддержку через Telegram-группу

И когда оператор может:
- находить пользователей
- смотреть баланс и историю
- корректировать баланс с комментарием
- вручную выдавать подписки
- блокировать пользователей
- обрабатывать заявки на вывод
- делать рассылки
- смотреть базовую статистику
