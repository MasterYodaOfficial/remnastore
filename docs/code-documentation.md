# Правила документации кода

## Назначение

Этот документ задает минимальный стандарт документации кода для `RemnaStore`.

Цель не в том, чтобы комментировать каждую строку, а в том, чтобы:

- быстро объяснять сложную бизнес-логику
- снижать риск поломок в деньгах, аутентификации, связке аккаунтов и подписках
- помогать новым разработчикам безопасно менять код
- не плодить воду и устаревающие комментарии

## Главный принцип

Документируем не синтаксис, а смысл, ограничения и риски изменения.

Если код и так очевиден, комментарий не нужен.

Если без пояснения разработчик может:

- неправильно понять бизнес-правило
- сломать идемпотентность
- продублировать денежный побочный эффект
- сломать аутентификацию или сценарий связки аккаунтов
- нарушить контракт между фронтендом, ботом и бэкендом

тогда документация обязательна.

## Что документировать обязательно

- публичные Python сервисные функции
- FastAPI-эндпоинты с неочевидным поведением
- сценарии оплаты
- сценарии связки аккаунтов
- `ledger` и все изменения баланса
- логика реферальных начислений
- обработка заявок на вывод
- обработка `webhook`
- фоновые задачи
- чувствительный с точки зрения безопасности код
- логика повторных попыток, fallback и идемпотентности
- сложные фронтенд helper-функции и оркестрация клиентских сценариев
- модули, где есть важные доменные инварианты

## Что обычно не нужно документировать

- очевидные getter/setter-подобные функции
- простые React-компоненты без сложной логики
- код, где комментарий просто повторяет строку кода
- тривиальные ветки `if/else`
- pydantic/schema/model поля, если имя и так самодостаточно
- технические детали, которые ясны из типов, имен и структуры кода

## Что должно быть в хорошем docstring

Хороший docstring обычно отвечает на такие вопросы:

- зачем существует эта функция или модуль
- какие у него побочные эффекты
- какие есть инварианты и ограничения
- что считается успешным результатом
- можно ли вызывать повторно
- как ведет себя код при ошибках, ретраях и повторной доставке событий
- какие внешние системы или контракты он затрагивает

## Общие правила

- писать коротко и по делу
- объяснять бизнес-смысл, а не Python или TypeScript синтаксис
- не дублировать название функции в виде бессмысленной фразы
- не писать комментарии, которые устареют быстрее кода
- использовать русский язык в docstring и кодовых комментариях для единообразия в проекте
- писать комментарии только там, где они реально уменьшают стоимость чтения и изменения кода

## Python

### Где docstring обязателен

- публичные сервисные функции
- нестандартные utility-функции со сложным поведением
- обработчики эндпоинтов с важными побочными эффектами
- классы исключений, если их назначение не очевидно из имени
- модули, которые управляют критичным сценарием

### Где docstring обычно не нужен

- маленькие private helper-функции с очевидной логикой
- `dataclass`/`pydantic` модели без нетривиального поведения
- тестовые helper-методы, если они ясны из контекста

### Рекомендуемый стиль

- 1 короткая summary-строка
- дальше 1 абзац с деталями только если они реально важны
- без лишнего формализма, если функция маленькая
- если есть деньги, аутентификация, внешние `callback` или идемпотентность, писать об этом явно

### Пример хорошего docstring

```python
def finalize_payment(...) -> PaymentResult:
    """Финализировать подтвержденный платеж провайдера и применить локальные эффекты.

    Создает локальное событие платежа, обновляет его статус и применяет
    пополнение `ledger` или финализацию покупки ровно один раз через `idempotency key`.

    Функция может вызываться повторно для одного и того же события провайдера.
    Повторные вызовы должны оставаться безопасными и не дублировать денежные эффекты.
    """
```

### Пример плохого docstring

```python
def finalize_payment(...) -> PaymentResult:
    """Финализировать платеж."""
```

Такой docstring почти бесполезен, потому что не объясняет ограничения и риски.

## FastAPI-эндпоинты

Эндпоинт стоит документировать, если:

- он инициирует деньги, подписку, связку аккаунтов или административный побочный эффект
- его поведение зависит от идемпотентности, повторной доставки `webhook` или внешнего статуса
- он не просто отдает список данных

### Что важно указать

- что делает эндпоинт
- мутирует ли он состояние
- что эндпоинт не делает напрямую, если это важно для архитектуры
- где происходит финальная бизнес-логика

### Пример

```python
@router.post("/payments/yookassa/topup")
async def create_topup(...):
    """Создать платежную попытку пополнения кошелька для текущего аккаунта.

    Эндпоинт создает попытку оплаты у провайдера, но не меняет баланс аккаунта
    напрямую. Баланс изменяется только после финализации подтвержденного
    `callback` от провайдера.
    """
```

## Модульные комментарии

Если модуль управляет критичным сценарием, в начале файла допустим короткий module docstring.

Он нужен, когда модуль:

- является единственной точкой важной бизнес-логики
- содержит правила идемпотентности
- оркестрирует внешние интеграции
- опасен для случайного рефакторинга

### Пример

```python
"""Оркестрация `callback` провайдера и локальной финализации платежей.

Модуль отвечает за идемпотентные переходы состояния платежа и должен
оставаться единственным местом, где подтвержденные платежные события
порождают биллинговые побочные эффекты.
"""
```

## Инлайн-комментарии

Инлайн-комментарии нужны редко.

Используем их только когда надо объяснить:

- почему выбран именно такой порядок действий
- почему нельзя упростить участок кода
- какой риск защищает данная проверка
- почему на первый взгляд странный код на самом деле обязателен

### Пример хорошего комментария

```python
# Сначала сохраняем событие провайдера, а уже потом применяем бизнес-эффекты,
# чтобы повторная доставка `webhook` не могла зачислить деньги дважды.
```

### Примеры плохих комментариев

```python
# Включаем состояние загрузки
loading = True
```

```python
# Проверяем, существует ли пользователь
if user is not None:
```

## Фронтенд

Во фронтенде комментариев должно быть меньше, чем в бэкенде.

Документировать стоит только:

- сценарий связки аккаунтов
- поведение, специфичное для Telegram Mini App
- сохранение и восстановление состояния незавершенного платежа
- синхронизацию UI со снимком состояния от бэкенда
- неочевидные ограничения, связанные с аутентификацией, повторными попытками или UX-согласованностью

Не нужно комментировать:

- обычную верстку
- простые presentation-компоненты
- тривиальные обновления состояния

## Бизнес-критичные зоны проекта

В этих местах документация особенно важна:

- `payments`
- `ledger`
- `purchases`
- `subscriptions`
- `account_linking`
- `withdrawals`
- `referrals`
- `notifications`
- `auth`
- `Telegram WebApp initData validation`
- `admin`-действия, влияющие на деньги, статус аккаунта и выдачу подписки

## Проверка качества документации

Перед добавлением docstring или комментария проверь:

- помогает ли он безопасно менять код
- объясняет ли он то, что неочевидно из имени и типов
- не устареет ли он сразу после следующего рефакторинга
- не дублирует ли он очевидное
- станет ли новому разработчику проще понять риск изменения

Если ответ в основном отрицательный, комментарий лучше не писать.

## Короткий стандарт для команды

Используем простое правило:

- документируем деньги, аутентификацию, связку аккаунтов, подписки, `webhook` и нестандартные побочные эффекты
- не документируем очевидное
- пишем коротко
- объясняем риски и инварианты
- предпочитаем хороший docstring плохому длинному комментарию
